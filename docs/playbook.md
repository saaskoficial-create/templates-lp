# Playbook de Templates de Landing Pages SaaS

## Objetivo
Este repositorio e a biblioteca principal da agencia para criar, demonstrar e adaptar landing pages de software. O v1 foi desenhado como `biblioteca + showcase`, nao como fabrica completa de LPs de clientes dentro do mesmo repo.

## Como escolher o template certo
### 1. SaaS BR Trial
Use quando o produto vende melhor por teste gratis, diagnostico ou demo curta.

Sinais de encaixe:
- SaaS para PMEs
- dor clara de operacao ou vendas
- onboarding simples
- CTA forte para trial

### 2. SaaS BR Venda Direta
Use quando a compra pode acontecer sem ciclo consultivo longo.

Sinais de encaixe:
- ticket mais acessivel
- oferta visivel
- comparativos de economia ou ROI
- pagina mais longa e comercial

### 3. SaaS Consultivo B2B
Use quando o produto exige alinhamento, implantacao ou integracoes.

Sinais de encaixe:
- venda mid-market ou enterprise
- decisor mais senior
- forte necessidade de credibilidade
- CTA para agendar demonstracao

### 4. Fintech / Infra / White Label
Use quando confianca, robustez e estrutura tecnica sao parte central da venda.

Sinais de encaixe:
- APIs, pagamentos, BaaS, subadquirencia
- compliance, liquidação, settlement
- integracoes complexas
- conversa tecnico-comercial

### 5. Category Leader
Use quando o produto precisa parecer amplo, consolidado e lider de categoria.

Sinais de encaixe:
- multiplos casos de uso
- varios times usando a plataforma
- narrativa mais aspiracional
- foco em marca e amplitude

## Como preencher copy por secao
- Comece pelo `conversionGoal` do template. Ele manda no tom do hero, no CTA e no fechamento.
- Preencha primeiro a `headline`, depois o `supportingCopy`, e so depois os bullets.
- Escreva beneficios como resultado de negocio, nao como lista fria de features.
- Se uma secao nao tiver prova real, reduza a promessa. Nunca invente numero.
- Mantenha linguagem de mercado do cliente. O template nao deve soar generico.

## Como pedir mockups e assets ao cliente
Solicite sempre ativos do proprio software. Ordem de prioridade:

1. Dashboard principal com dados plausiveis
2. Prints de modulos centrais
3. Fluxos reais do sistema
4. Integracoes ou mapas de dados
5. Video curto de navegacao
6. Fotos reais do time, especialistas ou cliente

Evitar:
- stock genérico sorrindo para notebook
- dashboard vazio
- mockup fake sem relacao com o produto
- video sem contexto funcional

Checklist de briefing visual:
- tela inicial
- modulo principal
- fluxo mais valioso
- integracoes relevantes
- numeros ou indicadores visiveis
- video curto, se houver

## Como adaptar para um SaaS real
- Escolha o template pelo modelo comercial, nao pela cor ou estilo.
- Duplique a definicao do template e substitua copy, inputs e guias de acordo com o nicho.
- Preserve a estrutura se ela ainda servir ao objetivo de conversao.
- Ajuste o tema visual depois da narrativa estar clara.
- Se o produto fugir muito dos 5 formatos, documente o motivo antes de criar um novo template.

## Consistencia visual e SEO
- Cada template pode ter identidade propria, mas deve usar a mesma base arquitetural.
- Hero sempre precisa comunicar valor acima da dobra.
- O heading hierarchy deve seguir `h1 > h2 > h3`.
- Toda imagem precisa nascer com `altSuggestion`.
- Prefira textos, capturas e blocos que ajudem a ranquear para termos do nicho do cliente.
- O SEO do v1 depende de metadata por rota, estrutura semantica e boa performance inicial.

## Regra para novos templates
Nao criar um sexto template apenas por vontade estetica. Um novo modelo so entra quando o mesmo tipo de briefing aparecer repetidamente e nao encaixar bem nos cinco atuais.
